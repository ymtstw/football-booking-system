# Release Gate の P0（本リポジトリ上の定義）

試験台帳の行 **仕様ID = `P0`**（機能 **Release_Gate**）は、**1 本の画面操作テストではない**。  
MVP 試験が一通り出た**あと**、本番・ステージングに乗せる**前の「運用・環境の総合ゲート」**を満たしたかをまとめて扱う行である。

## P0 という用語

- ここでいう **P0** は「最優先でリリース前に満たす**環境・運用チェック**」のラベルである。
- 元の Excel ブックでは **シート `Release_Gate` の行 2〜25 付近**に、**優先度列が `P0`** の行が並び、`MVP_Minimum_Run` の **ID `P0`** 行の式がそこ（例: `Release_Gate!$D$2:$D$25`）と突き合わせる想定である。
- CSV だけをエクスポートした場合、`Release_Gate` シートの**本文が同梱されない**ため、このドキュメントで**観点**を補完する。

## 合否（台帳の P0 行の意味）

- **PASS**: 下記**観点**を、対象環境（少なくとも **staging または本番想定**）で**手動**により確認し、**問題なし**と判断できる。
- 個別の機能 TC（`RSV-001` など）の代替にはならない。あくまで**併用**。

## 確認観点（MVP_Minimum_Run リリースゲート行のメモと同旨）

`docs/qa/MVP_Minimum_Run.csv` の **環境/運用前提 → リリースゲート → P0** 行のメモにある次の区分を、**行単位のチェックリスト**として扱う。

| 観点 | 確認内容の例（目安） |
|------|----------------------|
| **環境変数** | 本番/ステージングの `.env` / ホスト設定に、URL・キー・秘匿が漏れなく設定されている。本番のダミー・未設定がない。 |
| **Cron** | Vercel（または想定スケジューラ）上の **Cron URL**、**`CRON_SECRET`（Bearer）**、対象ルート（締切ロック・マッチング・通知など）が設計どおり。手動 or `npm run test:staging` 等で到達性・401/503 の挙動を確認。 |
| **Supabase** | 本番プロジェクト接続、マイグレーション適用、**RLS / ポリシー**、Service Role・Anon の使い分け。 |
| **Resend** | 本番/ステージでメール送達。ドメイン・API キー・差出人。テスト送信で届くこと。 |
| **前日運用** | **前日最終バッチ（16:30 JST 頃・利用者向け目安 17:30 まで）** を想定した **JOB03** や、運用手順（誰がいつ Cron をみるか等）が合意・実行可能。 |
| **スモーク** | 本番 or **staging** で**最小経路**が**ブラウザ**（＋下記 API 表）で通る。詳細は **「P0-06 スモーク：具体手順」**。 |
| **セキュリティ** | 個人情報の取り扱い、**ログに秘匿の載せない**、**HTTPS**、管理画面の**認可**（`AL-001`〜`003` 系）が運用方針と合う。 |

上表は**カテゴリの要約**である。Excel の `Release_Gate` シートに**より細かい 1 行 1 行**がある場合は、**そちらを正**とし、すべて **完了**（および結果列が **PASS** 等）になったとき、台帳の **P0 行**を**合格**とする。

## P0-06 スモーク：具体手順（本番 / staging 用）

表の**スモーク**は「主要機能の網羅的 E2E」では**ない**。**デプロイ直後の致命障害**（全周 5xx、白画面、管理に一切入れない、公開 API が常時壊れている、等）にすぐ気づくための、**10〜20 分程度**の**なぞり**と割り切る。

- **スキップ**（例: 受付中の開催日が 1 日もない）はよいが、**理由は試験記録に必ず残す**。
- **深い枠**（境界値の網羅、二重送信、`RSV-001` など）は、別行の仕様 ID で扱う。

### 1. 対象 URL ・ 証跡

- **staging または本番**の先頭（例: `.env.staging` の `STAGING_BASE_URL`、本番のドメイン）を用いる。
- 台帳には **実施日時**と、使った **base URL** を残す。

### 2. 公開（予約導線）— ブラウザ

1. トップ **`/`** — 200 で、案内・導線が**表示**される。  
2. 予約 — 例: **`/reserve` / `reserve/calendar`** 等から、**1 日分**の**日付付き予約画面**（**`/reserve/YYYY-MM-DD`**) へ辿り着く。枠・昼食 UI が、**致命的に壊れていない**こと。  
3. **予約の完走**（201 まで）は **P0-06 では必須にしない**（中身は `RSV-010` 等の別行）。受付中が無い場合は「開催日なし」と記録する。  
4. 任意: 有効な予約トークンがあれば該当画面。無ければ省略可。

### 3. 管理 — ブラウザ

1. **`/admin/login`** — `app_admins` の**実アカウント**でログインできる。  
2. ログイン後、**`/admin/dashboard`** 系が 200 かつ**白画面でない**、**主要エラー（5xx 連打）でない**。  
3. 未認証のリダイレクト、非管理者の拒否等は `AL-001`〜`003` と**重なり可**（二重記入でよい）。

### 4. 主要 API（最小。staging では `npm run test:staging` と一致）

同一 base について、**`npm run test:staging`**（`.env.staging` に `STAGING_BASE_URL` 等。Cookie は任意）**または** curl / DevTools で、少なくとも次の**意図**を満たす（`tests/staging/staging-smoke.staging.test.ts` と同じ方針）。

| 操作 | 期待 | 台帳の例（仕様ID） |
|------|------|---------------------|
| `GET /api/event-days` | 200 | API-ED-001 |
| `GET /api/event-days/not-a-date/availability` | 400 | API-AV-001 |
| `GET /api/event-days/1900-01-01/availability` | 404 | API-AV-001 |
| `GET /api/cron/lock-event-days` ＋ **誤** Bearer | 401 または 503 | CK-001 |
| 未認証 `GET /api/admin/notifications?status=failed` | 401 | MVP-NOTIF-401 |
| 未認証 `POST /api/admin/matching/undo`（`eventDayId` を入れた JSON 等） | 401 | TC-EX-UN-401 |

- **任意**: `STAGING_ADMIN_COOKIE` があると **`GET /api/admin/dashboard/next-event-day`** 系（`MVP-DASH-400/200`）の一部を Vitest 側で追加検証可。

**注意**: 上表は P0-06 用の**短いリスト**に過ぎず、結合テスト全体（40 本超）の代替ではない。
