# ドラフト要件: 予約あり開催日の強制枠変更

**状態:** ドラフト（実装・テストの正本は未確定。確定時は `implemented-behavior-catalog.md` 等へ反映する）

**対象:** 管理画面の **強制枠変更**（予約が 1 件以上 `active` な `event_day` に対する枠の変更）。通常ルート（予約ゼロ時の枠編集）は本ドラフトの対象外。

---

## 1. 背景と問題

強制枠変更は **枠マスタの数値変更ではなく、既存予約の再配置（または不整合リスク）** を伴う。保存前に **変更後の「どの枠にどの予約が乗るか」** が見えないまま確定させると、誤操作・説明責任・障害時の再現が困難になる。

---

## 2. 目的

- 運用者が **保存前に結果を理解したうえで** 確定できること。
- 影響チームへ **変更内容を通知可能**であること（送信失敗時も追えること）。
- **誰がいつ何をしたか**を後から説明できること。

---

## 3. 要件（必須）

### 3.1 変更後割当の可視化（必須）

**予約が存在する状態での強制枠変更では、保存（確定）前に、変更後の各枠への予約の割当結果を一覧表示すること。**

一覧に含める最低限の情報（列の例）:

- 枠: 識別子または表示名（時刻・枠コード等、運用で迷わないキー）
- 予約: チーム名、代表者名（または連絡先の一部）、予約 ID または管理画面への参照に足るキー

**禁止:** 上記一覧なしで強制枠変更の保存のみを許す UI／API 挙動（プレビュー API と確定 API を分ける場合は、確定前にプレビュー結果の取得を必須とする）。

### 3.2 確定前の確認フロー（必須）

「同じ文言の OK を 2 回」だけの二重化は要件としない。**確認する内容を分ける**こと。

最低 3 段階（画面または明確なステップ）:

1. **変更内容プレビュー**  
   変更後の「枠 × 予約」一覧（読み取り専用でよい）。
2. **通知対象・影響範囲の確認**  
   誰にメール（通知）を送るか、件数、送信タイミング（§4）の説明を表示する。
3. **最終確定**  
   ここでの操作のみが DB 更新および通知キュー投入を行う。

### 3.3 重大操作の明示（必須）

強制枠変更の導線では、**予約の再配置を伴う**旨を常時明示する（バナー・見出し等）。最終確定前に **短い警告文の再掲**を行う。

---

## 4. 通知（メール／キュー）

### 4.1 誰に送るか（要件で固定）

**影響を受けた予約の代表者（連絡先メール）全員**を正とする（「枠時刻のみ変更で slot 紐づけ不変」の予約を含めるか否かは **別途フラグまたは仕様一行**で固定する。未決のまま実装しない）。

### 4.2 いつ送るか

**保存確定後**に、既存の **通知キュー**（例: `notifications` 行を `pending` で INSERT）に積む。同期で Resend まで完了を待たないことを許容する。

### 4.3 失敗時

送信失敗は **`notifications.status = failed`** 等で追跡可能とし、**管理画面から再送可能**であること（既存の失敗通知・再送導線があればそれに合わせる）。

---

## 5. 監査ログ（強く推奨／MVP では最低限でも可）

メール履歴だけに依存しない。**操作そのもの**を記録する。

最低限記録する項目:

| 項目 | 例 |
|------|-----|
| 管理者 | 実行者の識別子（ユーザー ID 等） |
| 実行日時 | `timestamptz` |
| 対象開催日 | `event_day_id` / `event_date` |
| 変更前の枠構成 | JSON または正規化されたスナップショット |
| 変更後の枠構成 | 同上 |
| 影響予約数 | 整数 |
| 通知対象数 | 整数（メールキューに積んだ件数） |

**任意だが有用:** 操作に 1 対 1 の **相関 ID**（プレビュー〜確定〜通知行を突合するため）。

---

## 6. スコープ外（本ドラフトでは決めない）

- 予約ゼロ時の通常枠編集 UI の全面改修。
- 参加者向けの SMS / LINE。
- 自動ロールバック（「直前の枠構成に戻す」単発ボタン）は将来検討。

---

- 仕様正本への取り込み: 確定後 `docs/spec/implemented-behavior-catalog.md` の管理 API／画面節に要約 + テスト ID。
